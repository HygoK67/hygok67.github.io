Thinking 4.4

C选项是正确的，fork 只在父进程中被调用了一次，在父进程正常产生返回值；而在子进程中，其从一开始就处在 fork 调用结束并且返回值设置为 0 的状态，并没有真正执行 fork 代码。

Thinking 4.5

对于虚拟地址在 USTACKTOP 以下的部分，每个进程都不同，因此在 fork 的时候，父进程希望子进程能够完整的继承自己所有的状态，需要复制这些页面。

对于 USTACKTOP 到 UXSTACKTOP 这一部分，包含了用户异常处理函数所要用到的栈的空间，这一部分在父子进程之间并不需要相同，因此不用映射。

对于 UXSTACKTOP 到 ULIM 这一部分，包含了内核提供的供用户程序进行读取的进程页表、页控制快、进程控制块等信息。这些内容对于用户进程是只读的，而且在内核分配进程控制块 env_alloc 的时候就已经完成了相关的映射，所以用户态的 fork 函数也可以不管这一部分的内存。

综上，duppage 函数需要负责映射的只有虚拟地址小于 USTACKTOP 的页面。

在某一个进程初始化 env_alloc() 的时候，将当前进程从 vpt 开始的 4M 空间所对应的页目录项从自己映射到了自己。因此在访问这 4M 空间的时候，先根据页目录索引查询页目录项得到的二级页表仍是页目录本身，随后再根据页表索引查询二级页表（页目录自己）便能够直接得到对应的二级页表所在的物理页面。相当于用户进程在访问这 4M 空间内的时候，就可以直接获取到对应的二级页表的数据，而并不是根据二级页表的数据再去访问其他的物理页面。

值得注意的是，我们并不希望用户进程具有更改页目录/页表的能力，所以在用户空间中访问这些页表所在的物理页的时候，应该是只读的。这通过修改自映射的那条页目录项的可写权限位 PTE_D 为 0 来做到。

Thinking 4.7

在 lab4 实验实现的用户态 cow_entry() 页写入异常处理函数所有的操作都在异常处理栈上进行，异常处理栈所在的页面不会被 duppage() 函数处理，因此不会产生页写入异常，不会产生异常的重入。但如果用户态的异常处理函数希望对处理的页面进行一些写入操作，那么可能就又会产生一次页写入异常，又会进入到内核所在的 do_tlb_mod() 函数中，此时Trapframe的$sp` 寄存器所在位置处在用户异常处理栈中，进行了异常重入。

在用户态执行处理函数时，无法访问到内核栈的数据，因此需要将发生异常的现场 Trapframe 拷贝到用户空间去以便用户异常处理程序进行读取。

Thinking 4.8

在用户态处理页写入异常降低了某一个程序对内核的占用时间，使得内核能够处理其他进程的相关事务。并且在用户态处理灵活性较高，用户程序可以根据自身的需要来决定某个出现异常的页面的各种处理方法，而内核则相较而言难做到这一点。

Thinking 4.9

假设 exofork() 的调用发生在 syscall_set_tlb_mod_entry() 的调用之前，则因为 fork() 函数运行时所在的栈并非是异常处理栈，而是普通的用户程序栈，其所在的页面有可能是之前的 fork() 产生的 COW 页面，此时当 exofork() 返回后，由于需要对 child 局部变量进行写入，便会产生 tlb mod 异常，

但此时尚未注册处理函数，于是内核会发现内核态的 do_tlb_mod() 处理函数找不到对应的用户处理函数，程序便崩溃了。

syscall_set_tlb_mod_entry() 函数的调用也肯定不能放置在写时复制保护机制完成之后，在调用 duppage() 函数的过程中也涉及到对普通用户栈的页面进行写的操作，该页面也有可能是 COW 页面，此时也需要页写入异常处理函数 cow_entry() 进行处理。

Thinking 3.1

首先，这句语句的目的是为了让每个用户进程都能够在kuseg空间下以只读方式访问该进程的页表。而为了实现这一点，就需要对这个进程的页目录（即 e->env_pgdir）进行修改，使其把对应的 kuseg 中的虚拟地址映射到存放真实的页表的物理地址去。

考虑一下访存的过程：假设用户对于某一个页表在 kuseg 里的虚拟地址发起了访存请求，此时会找到页目录，以该虚拟地址的前十位作为索引，在页目录中找到一个二级页表，然后再以虚拟地址的接下来10位作为页表的偏移量，在这个二级页表里面找到对应的物理页。而我们希望这个找到的物理页就是二级页表，因此刚才映射到这个物理页的那个二级页表就应该是该进程的页目录，也就是说，在映射把进程的页表映射到用户空间时，我们需要通过页目录来找到页目录！

映射的页表在用户空间中占用的总大小为 4MB，其地址前十位不变化，因此页目录中的索引不发生变化，为 PDX(UVPT)，只需要一个页目录项。我们希望该页目录项能够映射回页目录，因此该页目录项前20位为页目录的物理页框号，即 PADDR(e->env_pgdir)，并且别忘了在标记位上将该项设为有效。由此，便得到了最终的页目录项

难点分析

在这次lab3的学习过程中，我主要遇到了以下这些问题：

有关物理内存大小与page2kva函数的有效范围的问题

在Lab3中，需要我们补全load_icode_mapper函数，以便能够在加载程序的二进制镜像时可以创建物理页并且把相应的数据拷贝至新增的物理页。
该过程共分为三步

  1. 分配一个新的物理页，获得其页控制块
  2. 复制所需要的数据至新分配的物理页
  3. 将给定的虚拟地址到该物理页的映射写入当前进程的二级页表系统中

我的疑问出现在第二步：

框架代码提示我们应该使用 memcpy 来进行内存内容的拷贝。memcpy 函数是利用指针来进行实现的，循环将源地址指向的数据拷贝到目标地址指向的空间中。同时又考虑到在mips 4KC cpu中，由于 CPU 发出的地址都是虚拟地址，因此所有 C 语言以及汇编语句中出现的相关地址都是虚拟地址。那么为了把数据拷贝到新的物理页面中去，我们就需要获知这个物理页面的虚拟地址。这里就出现了问题，即然页表的写入还未完成，那么如何确定其虚拟地址？

正确的操作应该是先获取该物理页面的物理地址，然后直接转换为内核态虚拟地址（这个转换不需要页表参与，是简单的线性映射），这样 CPU 就能够对该页面进行读写。我当时在这里产生疑问的主要原因是因为我认为这里的物理页只能够由用户态的虚拟地址访问到，而不能被内核态虚拟地址访问到，但实际证明这个想法是不对的。

在实验中，MOS 操作系统可用的物理内存大小为 65536 KiB 也即 64 MiB，查看 mmu.h 可以知道 kseg0 空间起始于 0x80000000，大小共 512 MiB ，因此所有的物理内存其实在内核态都得到了映射，CPU可以通过发出内核态的虚拟地址来访问任何的物理内存，这样，就相当于 CPU 直接对物理内存进行了读写。同时也因为 kseg0 大小的限制，MOS 操作系统的物理内存上限为 512 MiB，超出该部分的内存不在 kseg0 的映射范围内，而 CPU 又无法直接发出物理地址，因此无法利用到这一部分内存。

有关进程调度相关的问题

Thinking 2.4

虚拟内存这个概念是相对于进程而言的，不同的进程应该有相互独立的虚拟内存。并且由于内存共享等需求，有可能会出现不同进程的相同虚拟地址映射到不同的物理地址以及不同进程的不同虚拟地址映射到同一物理地址的情况。为了解决多进程映射的这个问题，只通过虚页号是不够的，就必须引入能够标识当前虚拟地址所在地址空间的ASID。如果TLB中没有ASID的话，那么所有的进程在查询TLB时其实都是在查询一张页表，虚拟内存就失去了意义，相当于整个虚拟地址空间变成了和物理空间的一对一双射，进程之间的独立性也不存在了。因此mips 4KC的TLB中含有ASID字段是必要的

ASID段共8位，无预留，故总共可有 $2^8=256$​ 个不同的地址空间，但是由于TLB中还可以设置 G 位，绕开ASID，所以其实最终的地址空间共有 257 个

Thinking 2.6

对于x86体系架构来说，其和mips的差别主要体现在两点，其一在x86体系架构上由逻辑地址转换为物理地址的过程中中间会经历一步线性地址。逻辑地址首先会经过分段单元进行处理。每个分段包含一个起始地址和一个长度，用来定义一个内存段。通过段选择子和偏移量，逻辑地址被映射到一个线性地址。然后再由分页机制对该线性地址进行变换，得到物理地址。第二个不同点就出在变换到物理地址的过程中，在mips中，页表的维护、地址的变换以及tlb的填写都由软件来完成，而在x86体系结构中，tlb的填写以及地址的变换都由mmu硬件来完成，软件只需要对页表进行维护即可。
